# AI 代码生成与修复标准

本文档用于约束本项目中的 AI 代码生成、问题修复、重构和验证方式。它不是 `WORKFLOW.md` 的替代品，而是 `WORKFLOW.md` 之后、具体源代码修改之前必须读取的项目级细则。

## 1. 文档目的与适用范围

适用范围：

- 需求开发、Bug 修复、重构优化、配置调整和文档更新。
- 前端 `Vue 3 + Vite`、后端 `Java 25 + Spring Boot + Maven`、WebSocket、数据库、测试和项目文档。
- AI 在同一模块反复出错时的规则沉淀。

核心目标：

- 让每次修改先理解项目规则，再动手改文件。
- 把历史错误转换成可执行的检查项。
- 让 Bug 修复同时带来可复用的预防规则。

## 2. 修改前必须读取的上下文顺序

每次任务开始时，必须按以下顺序读取：

1. `WORKFLOW.md`
2. `AI_CODE_GUIDE.md`
3. 当前需求文档：`需求文档/需求文档vX.X.X.X-*.md`
4. 与任务直接相关的源代码、测试、配置和文档

读取后必须在回复或完成报告中说明：

- 本次遵循的关键标准。
- 本次涉及的高风险规则。
- 本次明确不触碰的非范围内容。

如果 `WORKFLOW.md`、`AGENTS.md`、需求文档和本文档存在冲突，优先级为：

1. 用户当前明确指令。
2. 安全、确认门禁和不可破坏性规则。
3. `WORKFLOW.md`。
4. 当前需求文档。
5. `AI_CODE_GUIDE.md`。

## 3. 通用代码生成原则

- 先读再改：修改已有文件前必须先读取当前内容，理解周边模式。
- 最小变更：只改为完成当前需求必需的文件和代码。
- 保持风格：命名、拆分方式、异常处理、返回结构、CSS 组织方式必须贴近已有实现。
- 不做顺手重构：除非需求明确要求，否则不重排无关代码、不统一无关格式、不移动文件。
- 不硬编码：魔术数字、URL、路径、枚举值、业务状态应优先使用现有常量、配置或集中定义。
- 不覆盖用户改动：看到未提交改动时，默认视为用户工作，必须围绕它们工作。
- 不把猜测当事实：不确定的业务规则必须从代码、文档、测试或用户确认中获得依据。
- 修改后要验证：规则沉淀不能替代构建、测试、冒烟和回归检查。

## 4. 前端 Vue/Vite 编码与样式规则

前端目录重点：

- 页面：`poker-frontend/src/views/**`
- 状态：`poker-frontend/src/store/**`
- 全局样式：`poker-frontend/src/style.css`
- 入口：`poker-frontend/src/main.js`

页面模块通常拆分为：

- `index.vue`
- 同名 `.html`
- 同名 `.js`
- 同名 `.css`

修改前端页面时必须同时检查：

- 模板是否引用了脚本中存在的字段、计算属性和方法。
- 脚本状态是否与 store、WebSocket 消息和后端字段一致。
- 样式是否影响桌面、手机、平板和横屏布局。
- 弹窗、抽屉、按钮、提示文字是否存在溢出、遮挡或不可点击。

前端禁止：

- 只改 `.html` 不检查同模块 `.js`、`.css` 和 store。
- 只按桌面视口调样式，不检查移动端和横屏。
- 用大量新颜色、新阴影或新布局风格破坏现有三国主题。
- 为一个页面问题改全局样式，除非确认影响面并完成回归。

前端推荐：

- 页面交互优先复用现有 store 和已有事件流。
- 样式使用稳定尺寸、响应式约束和明确层级，避免动态内容撑坏布局。
- 修改 UI 后至少列出桌面端、手机端和横屏端的验证结论。
- 新增或调整武将技能时，必须同步检查并更新 `GameBoard` 交互、`Lobby` 更新公告、`RulesDetail` 详细规则和前端 store 消费路径；技能必须写在 `RulesDetail` 的“武将技能系统”章节，不得散落到基础规则、核心机制或锦囊章节。
- 技能语音统一通过仓库根目录 `generate_Yunxia.py` 生成，输出到 `poker-frontend/public/audios/` 后再在 `gameSocket.js` 或既有音频入口接入；技能发动语音只读技能名称，格式为“技能名！”，例如“归心！”。
- 过程截图、浏览器验证截图和临时视觉产物统一放入根目录 `temp-screenshots/`，不要散落到仓库根目录或业务源码目录。
- 大厅公告文案保持短句和同类聚合，例如“武将技能【苦肉】【铁骑】【归心】已上线。”；不要把技能完整规则、结算细节或长解释塞进公告。
- 详细规则文案优先保持现有条目的写法：主句说明核心效果，`rules-note` 只补必要限制或特殊结算。跨技能联动说明应集中写在新增/主责技能条目中，不要在多个已有技能条目里重复插入同一套解释。

## 5. 后端 Spring Boot/Java 编码规则

后端目录重点：

- Controller：`poker/src/main/java/com/game/poker/controller/**`
- Service：`poker/src/main/java/com/game/poker/service/**`
- WebSocket：`poker/src/main/java/com/game/poker/websocket/**`
- Model/DTO/Entity：`poker/src/main/java/com/game/poker/model/**`、`dto/**`、`entity/**`
- Repository：`poker/src/main/java/com/game/poker/repository/**`
- 测试：`poker/src/test/java/com/game/poker/**`

后端修改前必须确认：

- 入口参数、空值、非法状态和权限边界。
- Service 是否已有同类方法或业务约束。
- DTO 是否会泄露敏感字段。
- 事务边界和持久化行为是否受影响。
- H2 本地环境与 MySQL 生产环境是否存在差异。

后端禁止：

- 在 Controller 里堆业务逻辑，绕过已有 Service。
- 改 Entity 字段、Repository 查询语义或迁移脚本而不单独确认。
- 用宽泛 `catch` 吞掉异常后返回成功。
- 只修主路径，不处理空房间、断线玩家、AI、旁观者、重复请求等边界。

后端推荐：

- 优先在 Service 层维护业务不变量。
- Bug 修复优先补对应单元测试或最小复现场景。
- 修改共享模型时同步检查 WebSocket 广播、DTO 映射和前端消费字段。

## 6. WebSocket 与游戏状态同步规则

涉及游戏实时同步时必须检查：

- 消息类型、字段名、发送方、接收方是否与前后端一致。
- 刷新、重连、断线玩家、旁观者和 AI 玩家状态是否互不污染。
- 广播是否可能重复、丢失、乱序或覆盖更新。
- 游戏阶段切换是否保持原子性，尤其是技能、锦囊、插队响应、超时和结算。

禁止：

- 只改前端显示，不检查后端广播源。
- 只改后端状态，不检查前端 store 消费路径。
- 在多个地方手动拼接相同消息结构，导致字段漂移。

推荐：

- 先找现有消息构造和消费链路。
- 修改消息结构时同步更新前端 store、视图和相关测试。
- 对复杂状态变化保留最小、明确的日志或测试证据。

## 7. 数据库、实体、DTO、Repository 修改规则

数据库相关默认高风险。以下操作必须先单独确认：

- 新增、删除或修改 Entity 字段。
- 修改 Repository 查询语义。
- 修改 `poker/src/main/resources/db/migration/**`。
- 修改生产 MySQL、本地 H2 或 Redis session 行为。

如确认需要修改，必须说明：

- schema 或持久化行为变化。
- H2 与 MySQL 的兼容性。
- 回滚或恢复方案。
- DTO 是否仍避免泄露敏感字段。
- 旧数据、空值、默认值和迁移顺序。

## 8. 测试与验证最低要求

按改动范围选择验证：

- 文档：检查 UTF-8、Markdown 可读性、路径、版本号、章节完整性。
- 前端：执行 `npm run build`，并做关键视口冒烟检查。
- 后端：执行相关测试类或 `mvn test` / `./mvnw.cmd test`。
- WebSocket：验证连接、重连、消息发送、广播和状态恢复。
- 游戏规则：验证关键牌型、技能、锦囊、超时、结算和战绩。
- 数据库：验证本地 H2 与生产 MySQL 语义差异，必要时补迁移测试。

如果无法执行某项验证，完成报告必须说明原因、风险和替代方案。

新增错误模式时使用以下格式。

```md

- 根因：
- 以后禁止：
- 推荐做法：
- 修改前检查：
- 验证方式：
- 关联模块：
- 首次记录日期：
```

当前已沉淀的项目级错误模式：

### 错误模式：本地前端开发代理误连线上后端

- 触发场景：本地启动 `poker-frontend` 和 `poker` 后，测试新技能、新接口或 WebSocket 行为。
- 错误表现：前端页面打开的是 `localhost:5173`，但 `/api`、`/ws` 通过 Vite proxy 被转发到线上 `39.102.60.181:8080`，导致本地后端新增逻辑无法生效，表现为“前端按钮可见但技能/接口无效”。
- 根因：`vite.config.js` 的开发代理默认目标指向线上后端，而不是本地 `127.0.0.1:8080`。
- 以后禁止：本地联调和验证新后端功能时，默认代理到线上服务器。
- 推荐做法：Vite 开发代理默认使用 `http://127.0.0.1:8080`；如需临时连线上，必须显式设置 `VITE_DEV_BACKEND=http://39.102.60.181:8080`，并在验证报告中说明。
- 修改前检查：检查 `poker-frontend/vite.config.js` 的 `devBackend`、前端启动日志和浏览器 Network 中 `/api`、`/ws` 的实际目标。
- 验证方式：本地启动后端后确认 `netstat` 中 `8080` 是 `com.game.poker.PokerServerApplication`，前端重新启动后再测试新功能；必要时查看本地后端日志是否收到对应 WebSocket/API 请求。
- 首次记录日期：2026-05-01，修复【归心无法使用】时沉淀。

### 错误模式：中文文档读取未显式指定 UTF-8 导致乱码

- 触发场景：在 Windows PowerShell 中读取 `WORKFLOW.md`、需求文档或中文 Markdown。
- 错误表现：终端输出中文乱码，影响规则理解和需求复核。
- 根因：未显式设置 PowerShell 输出编码和 `Get-Content -Encoding UTF8`。
- 以后禁止：在未设置 UTF-8 输出的情况下直接依赖中文文档终端输出。
- 推荐做法：每个 PowerShell 会话第一次输出前设置 `$OutputEncoding = [Console]::OutputEncoding = [Text.UTF8Encoding]::new($false)`，读取中文文件时使用 `-Encoding UTF8`。
- 修改前检查：确认中文文档能正常显示。
- 验证方式：抽查目标 Markdown 的标题和关键章节是否无乱码。
- 关联模块：项目文档、需求文档、工作流。
- 首次记录日期：项目引入 `AI_CODE_GUIDE.md` 时。

### 错误模式：只接入普通 PASS 导致技能式要不起漏触发

- 触发场景：新增或修改依赖“要不起”“不出牌”“结束本次行动”的技能联动，例如【归心】监听他人要不起、【固守】主动发动后结束行动、【观星】在要不起后延迟结算。
- 错误表现：普通点击“要不起”能触发联动，但通过技能按钮达成的等价要不起不触发；前端看起来桌面牌仍属于目标玩家，后端却没有进入对应待响应状态。
- 根因：只在 `passTurn(...)` 中接入联动判断，漏掉 `useGushou(...)`、`resolveGuanxing(...)` 等也会推进回合、跳过出牌或免除/替代摸牌惩罚的旁路入口。
- 以后禁止：实现要不起联动时只搜索普通 `PASS` 消息或只修改 `passTurn(...)`。
- 推荐做法：先梳理所有会“结束当前行动/推进回合/清理桌面/免除要不起惩罚”的入口，把共享判断抽成 Service 方法并在每条等价路径调用。
- 修改前检查：核对 `passTurn(...)`、主动技能方法、锦囊响应、AI fallback、超时自动处理和断线处理是否都会影响 `lastPlayPlayerId`、`lastPlayedCards`、`currentTurnIndex`。
- 验证方式：为普通 PASS 和至少一个技能式要不起入口分别补单元测试；观察后端日志中 `lastPlayPlayerId`、技能禁用态和待响应状态是否一致。
- 关联模块：`GameService`、`GameWebSocketHandler`、技能单元测试。
- 首次记录日期：2026-05-01，修复【归心不因固守式要不起重置】时沉淀。

### 错误模式：挂起结算阶段仍接受普通回合动作导致桌面被滞后动作覆盖

- 触发场景：用户主动出牌、使用技能或进入觉醒/锦囊挂起结算的同时，前端倒计时自动出牌、自动要不起、Bot 行动或重复点击消息延迟到达。
- 错误表现：首个合法动作 A 已经结算，但滞后的自动动作 B 仍被服务端接受，导致手牌被二次移除、`lastPlayedCards` 被覆盖、回合被二次推进，前端桌面最终留下 B。
- 根因：普通回合入口只判断 `isPlayerTurn(...)`，没有同时判断 `currentAoeType` / `pendingAoePlayers` 是否已经进入挂起结算；部分技能入口也没有统一房间锁，无法保证同一行动窗口只提交一次。
- 以后禁止：只依赖前端 `isTimeoutTriggered`、按钮禁用或倒计时重置来防并发；在 `currentAoeType` 非空或 `pendingAoePlayers` 非空时继续允许 `PLAY_CARD`、`PASS`、普通技能按钮或 Bot 普通行动改变状态。
- 推荐做法：服务端普通回合动作入口统一使用房间级锁，并在锁内先执行挂起结算门禁；只有当前挂起阶段对应的处理器（如 `kurouAwakenDiscard`、`respondAoe`、`resolveWgfd`、`resolveGuixinDecision`）可以处理该阶段消息。
- 修改前检查：核对 `playCards(...)`、`passTurn(...)`、`replaceCard(...)`、`useLuanjian(...)`、`useGushou(...)`、`useKurou(...)`、`useGuixin(...)`、Bot 行动和 WebSocket 分发是否都不会绕过挂起阶段门禁。
- 验证方式：补测试模拟“主动动作 A 成功后滞后自动动作 B 到达”，断言 B 被拒绝/忽略，桌面牌、手牌、弃牌堆、回合索引和挂起状态仍保持 A 后的状态；再跑核心技能测试和小样本技能模拟。
- 关联模块：`GameService`、`GameWebSocketHandler`、`gameSocket.js`、`KurouSkillServiceTest`、技能/Bot 回归测试。
- 首次记录日期：2026-05-05，修复【出牌与倒计时并发冲突导致桌面留下自动牌】时沉淀。

### 错误模式：新增技能漏同步公告、详细规则和语音资产

- 触发场景：新增或调整武将技能、技能语音、技能按钮和技能结算文案。
- 错误表现：代码功能可用，但大厅公告、`RulesDetail` 详细规则、技能语音或前端播放分支仍停留在旧版本，玩家无法从界面理解最新规则。
- 根因：只关注后端结算和按钮可用性，未把规则页面、公告和音频资产列为技能开发的固定交付物。
- 以后禁止：新增技能时只改 `GameService` / `GameBoard`，不检查 `Lobby`、`RulesDetail`、`gameSocket.js` 和 `poker-frontend/public/audios/`。
- 推荐做法：技能需求文档和实现清单中固定列出“规则同步、公告同步、语音生成与接入、前后端构建验证”。
- 修改前检查：确认是否需要通过 `generate_Yunxia.py` 生成新语音，确认临时截图是否放入 `temp-screenshots/`。
- 验证方式：构建前端并检查 `SKILL_USED` 分支、音频文件路径、公告文本和详细规则文本都包含新技能。
- 关联模块：`Lobby`、`RulesDetail`、`gameSocket.js`、`poker-frontend/public/audios/`、`generate_Yunxia.py`。
- 首次记录日期：2026-05-02，落实【归心】语音与规则页面同步时沉淀。

### 错误模式：技能规则放错章节且语音文案过度发挥

- 触发场景：为新增武将技能补规则说明和技能发动语音。
- 错误表现：新增技能【归心】被写进 `RulesDetail` 的“基础卡牌与牌型”章节，导致不和其他武将技能排版在一起；语音文案写成完整台词“归心所向，众望归一！”，不符合项目现有技能语音只喊技能名的风格。
- 根因：没有先对照规则页现有信息架构和已有音频命名/台词模式，直接按功能点插入内容。
- 以后禁止：新增武将技能时把技能条目插到非“武将技能系统”章节；技能语音擅自扩写成口号、长句或剧情台词。
- 推荐做法：先提取用户已经修改过的文案和页面现有排序，再做最小移动；技能语音统一使用“技能名！”。
- 修改前检查：检查 `RulesDetail.html` 中技能条目的章节位置，检查 `generate_Yunxia.py` 的 `texts` 文案是否只包含技能名称。
- 验证方式：前端构建通过；人工检查详细规则页中新增技能与其他技能同属一个列表；确认生成的 `action_*.mp3` 来源文本为“技能名！”。
- 关联模块：`RulesDetail.html`、`generate_Yunxia.py`、`poker-frontend/public/audios/`、`gameSocket.js`。
- 首次记录日期：2026-05-02，修正【归心】规则位置和语音文案时沉淀。

### 错误模式：规则与公告文案过度展开，未沿用用户精简写法

- 触发场景：新增技能后同步大厅公告和 `RulesDetail` 详细规则。
- 错误表现：大厅公告把【归心】展开成多句规则解释；详细规则在【观星】【固守】条目中重复插入【归心】相关说明，使文案变散、信息重复。用户后续改为“武将技能【苦肉】【铁骑】【归心】已上线。”，并把归心联动集中到【归心】条目。
- 根因：没有先提取用户已有文案风格和页面信息密度，按需求文档内容直接铺开说明。
- 以后禁止：公告里写完整技能规则；为了说明新技能，把同一联动规则重复塞进多个既有技能条目。
- 推荐做法：公告只写短句和上线范围；详细规则以新增技能条目为主责位置，必要时一句话交代被影响技能，不改动已有技能条目的核心说明。
- 修改前检查：对照用户最近一次手改文案，确认是否短、直、聚合；检查同一规则是否在多个条目重复出现。
- 验证方式：人工通读 `Lobby` 更新公告和 `RulesDetail` 武将技能列表，确认公告不超过一句主要说明，详细规则没有重复解释。
- 关联模块：`Lobby.html`、`RulesDetail.html`、需求文档、公告文案。
- 首次记录日期：2026-05-02，用户修正【归心】公告与详细规则文案后沉淀。

## 10. Bug 修复复盘模板

每次 Bug 修复完成后，必须判断是否需要更新本文档。

满足任一条件时，应该沉淀规则：

- 同类错误已经重复发生。
- 本次问题暴露出已有规则不明确。
- 涉及容易被误判的业务规则。
- 涉及前后端状态一致性、WebSocket、数据库兼容性、响应式布局或测试遗漏。
- 修复过程中发现新的必读上下文文件。

复盘格式：

```md
## Bug 修复复盘

- 问题现象：
- 直接原因：
- 深层原因：
- 本次修复：
- 新增验证：
- 是否需要沉淀规则：
- 沉淀到 `AI_CODE_GUIDE.md` 的位置：
```

如果不沉淀，必须说明原因，例如“一次性环境问题”“用户输入问题”“已有规则已覆盖”。

## 11. 文档自迭代流程

每次 Bug 修复后的流程：

1. 完成最小修复。
2. 运行对应验证。
3. 判断是否有可复用错误模式。
4. 如有，更新 `AI_CODE_GUIDE.md` 的历史错误模式或对应规则章节。
5. 在完成报告中写明“规则沉淀”结果。

更新本文档时必须保持：

- 规则短小、明确、可检查。
- 优先补充已有章节，避免重复开新章节。
- 一次性问题不进入长期规则。
- 不用文档更新替代测试证据。

## 12. 禁止行为与高风险操作清单

除非用户明确确认，否则不得：

- 修改数据库 schema、迁移脚本或生产数据行为。
- 新增、升级或删除第三方依赖。
- 改变认证、授权、安全策略。
- 修改生产部署配置。
- 删除重要文件或清理大量本地文件。
- 大规模调整目录结构。
- 覆盖用户已有未提交改动。
- 跳过需求文档确认直接修改业务代码。

高风险操作必须在动手前说明：

- 影响范围。
- 兼容性风险。
- 回滚方式。
- 验证计划。
