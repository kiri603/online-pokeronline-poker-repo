# 项目更新工作流

本文档规定本项目每次需求开发、问题修复、重构、文档更新和验证交付的标准流程。


## 最重要

- 始终先生成并确认需求文档，再修改业务代码。
- 每次任务必须展示并更新执行检查清单。
- 不得跳过 Git 状态检查、需求文档、验证和完成报告。
- 不覆盖用户已有未提交改动。
- 不为完成某一步而扩大任务范围。
- 中文文档统一使用 UTF-8。
- 代码生成、问题修复和重构时，必须在读取本文件后读取 `AI_CODE_GUIDE.md`，并按其中的项目级代码标准执行。

## 执行检查清单

每个需求开发、问题修复、重构或文档任务都必须从以下检查清单开始，并在每一步完成后立刻把 `[ ]` 更新为 `[x]`。

```md
## 执行检查清单
- [ ] Step 0：任务分类 + Git 状态确认（说明当前已有改动，避免覆盖）
- [ ] Step 1：已读 `WORKFLOW.md` + `AI_CODE_GUIDE.md` + 相关文档，并完成需求文档撰写（路径：需求文档/需求文档v{版本号}-{更新名称}.md，粘贴关键摘要）
- [ ] Step 2：用户确认需求文档（确认后才允许修改源代码）
- [ ] Step 3：实现（说明改动范围和未触碰范围）
- [ ] Step 4：冒烟测试（浏览器、接口或日志证据）
- [ ] Step 5：编译 + 自动化测试（列出执行命令和结果）
- [ ] Step 6：项目专项回归检查（前端、后端、WebSocket、数据、编码）
- [ ] Step 7：完成报告（说明修改文件、验证结果、跳过项和风险）
```

不得在前一步未完成时开始后续步骤。

例外：

- 如果任务只是“制定计划”或“写技术文档”，不得修改业务源代码。
- 如果用户明确要求只分析，不得创建需求文档或修改文件，除非用户随后确认。
- 如果用户要求修复紧急故障，仍需先完成最小版需求文档，至少覆盖背景、目标、影响范围、验收标准和测试计划。

## Step 0：任务分类与工作区检查

开始前必须先分类任务：

| 任务类型 | 判定标准 | 默认动作 |
| --- | --- | --- |
| 需求开发 | 新增或改变功能行为 | 生成需求文档，确认后实现 |
| 问题修复 | 修复已有错误、异常、显示问题或交互问题 | 生成修复型需求文档，确认后修复 |
| 重构优化 | 改善结构、性能、可维护性但不改变行为 | 明确不变行为和回归范围 |
| 文档更新 | 只更新说明、规范、README、流程 | 不修改业务代码 |
| 配置/部署 | 修改构建、环境变量、部署、跨域、数据库配置 | 先确认风险，再执行 |

必须运行：

```bash
git status --short
```

检查时必须明确说明：

- 当前是否存在用户已有改动。
- 本次计划触碰哪些文件。
- 哪些已有改动必须保持不动。

当前仓库经常会出现大量本地截图、`.playwright-mcp/`、临时文件和本地数据库文件，默认不纳入任务范围，除非用户明确要求。后续过程截图和临时视觉验证产物统一保存到根目录 `temp-screenshots/`。

## Step 1：读取相关文档并生成需求文档

### 1.1 必读内容

所有任务默认先读：

- `WORKFLOW.md`
- `AI_CODE_GUIDE.md`
- 根目录 `README.md`
- 与任务相关的前端、后端、配置或测试文件

读取顺序必须为：

1. `WORKFLOW.md`
2. `AI_CODE_GUIDE.md`
3. 当前需求文档
4. 与任务相关的源代码、测试、配置或文档

如果当前任务只是在生成第一版需求文档，且还没有对应需求文档，则先读取 `WORKFLOW.md`、`AI_CODE_GUIDE.md` 和任务相关上下文，再生成需求文档。

读取后必须说明：

- 本次遵循的关键标准。
- 本次可能触碰的高风险规则。
- 本次明确不触碰的非范围内容。

按任务类型追加读取：

| 任务范围 | 重点读取 |
| --- | --- |
| 前端页面 | `poker-frontend/src/views/**`、相关 store、`poker-frontend/src/style.css` |
| 前端状态 | `poker-frontend/src/store/**` |
| 游戏桌面 | `GameBoard` 视图、`gameState.js`、`gameSocket.js`、后端 `GameService`、`RuleEngine` |
| 大厅/等待房间 | `Lobby`、`WaitingRoom`、`RoomController`、`GameService` |
| 登录/账号 | `Auth`、`authStore.js`、`AuthController`、`AuthService`、`UserService` |
| 好友/个人中心 | `Social`、`socialStore.js`、`SocialController`、`SocialService` |
| WebSocket | `GameWebSocketHandler`、`WebSocketConfig`、`gameSocket.js` |
| 数据库 | entity、repository、service、`poker/src/main/resources/db/migration/**` |
| 构建/依赖 | `poker-frontend/package.json`、`poker/pom.xml` |

### 1.2 需求文档位置

需求文档必须写入：

```text
需求文档/
```

文件名格式：

```text
需求文档v{版本号}-{更新名称}.md
```

示例：

```text
需求文档/需求文档v0.0.0.1-大厅房间列表优化.md
```

### 1.3 版本号规则

版本号使用四段数字：

```text
v主版本.次版本.修订版本.任务版本
```

规则：

1. 初始版本为 `v0.0.0.1`。
2. 每段数字最多两位，即 `0` 到 `99`。
3. 每次新需求默认递增最后一段。
4. 如果最后一段达到 `99`，向前一段进位。
5. 生成新文档前，必须扫描 `需求文档/` 中已有 `vX.X.X.X` 版本，取当前最大版本后递增。
6. 如果文件夹为空，使用 `v0.0.0.1`。
7. 每次回复必须明确写出本次生成的需求文档文件名和版本号。

### 1.4 需求文档模板

每份需求文档至少包含：

```md
# 需求文档v{版本号}：{更新名称}

## 1. 背景
说明问题来源、当前痛点、为什么需要更新。

## 2. 更新目标
列出本次更新必须达成的结果。

## 3. 任务类型
标明是需求开发、问题修复、重构优化、文档更新或配置/部署。

## 4. 影响范围
说明涉及的前端页面、后端服务、接口、WebSocket 消息、数据结构、配置或文档。

## 5. 功能需求
用明确条目描述新增、修改或删除的行为。

## 6. 非功能需求
包括性能、兼容性、响应式布局、错误处理、安全性、编码、可维护性等要求。

## 7. 不在本次范围
明确本次不做的内容，防止范围扩大。

## 8. 验收标准
每条标准必须可测试、可观察。

## 9. 风险与注意事项
列出可能影响现有功能的地方。

## 10. 测试计划
说明需要执行的自动化测试、构建、手动验证、浏览器验证和回归场景。
```

空文件、只有文件名、只有标题，都不算完成 Step 1。

Step 1 完成时必须同时具备：

- 已读文档摘要。
- 完整需求文档路径。
- 当前版本号。
- 对任务范围和非范围的明确说明。

## Step 2：用户确认需求文档

需求文档生成后，必须先展示给用户确认。

未经用户确认，不得进行以下操作：

- 修改源代码。
- 修改数据库迁移。
- 新增、删除或重命名文件。
- 安装、升级或删除依赖。
- 执行会改变仓库状态的代码生成、格式化或迁移命令。

用户确认方式可以是：

- 明确说“确认”“继续”“执行”“按这个做”。
- 明确回复 `y` 或 `c`。

如果用户修改需求，必须更新需求文档后再次确认。

## Step 3：实现

确认后按最小范围实现。

必须遵守：

- 保持现有项目结构和命名风格。
- 优先编辑现有文件，必要时才新增文件。
- 不做无关重构。
- 不做无关格式化。
- 不新增第三方依赖，除非用户确认。
- 不修改数据库 schema，除非用户确认。
- 不改变认证、授权、安全策略，除非用户确认。
- 不覆盖用户已有未提交改动。

### 3.1 前端项目特点

前端是 `Vue 3 + Vite`，主要结构：

- 入口：`poker-frontend/src/main.js`
- 根组件：`poker-frontend/src/App.vue`
- 全局样式：`poker-frontend/src/style.css`
- 状态管理：`poker-frontend/src/store/**`
- 页面模块：`poker-frontend/src/views/**`

页面模块通常按以下形式拆分：

- `index.vue` 或主 `.vue`
- 同名 `.html`
- 同名 `.js`
- 同名 `.css`

修改页面时必须同时检查相关的模板、脚本、样式和 store。

重点页面：

- `Auth`：登录、注册、游客入口、验证码。
- `Lobby`：大厅、房间列表、规则入口、公告。
- `WaitingRoom`：等待房间、准备、AI、旁观、邀请。
- `GameBoard`：游戏桌面、出牌、技能、锦囊、表情、回合状态。
- `RulesDetail`：规则详情。
- `Social`：个人中心、好友、消息。

新增或调整武将技能时，必须同步更新 `RulesDetail` 详细规则，并把技能条目放在“武将技能系统”章节；若技能对玩家可见，也必须同步检查 `Lobby` 更新公告和前端技能语音接入。技能发动语音统一使用“技能名！”格式，不写额外台词。

### 3.2 后端项目特点

后端是 `Java 25 + Spring Boot + Maven`，主要结构：

- 启动类：`poker/src/main/java/com/game/poker/PokerServerApplication.java`
- 控制器：`poker/src/main/java/com/game/poker/controller/**`
- 业务服务：`poker/src/main/java/com/game/poker/service/**`
- WebSocket：`poker/src/main/java/com/game/poker/websocket/**`
- 模型：`poker/src/main/java/com/game/poker/model/**`
- DTO：`poker/src/main/java/com/game/poker/dto/**`
- 实体：`poker/src/main/java/com/game/poker/entity/**`
- 仓库：`poker/src/main/java/com/game/poker/repository/**`
- 配置：`poker/src/main/java/com/game/poker/config/**`
- 测试：`poker/src/test/java/com/game/poker/**`

重点服务：

- `GameService`：房间、玩家、游戏流程、AI、状态广播。
- `RuleEngine`：牌型与出牌规则。
- `AuthService`：注册、登录、游客、会话。
- `SocialService`：好友、消息、邀请。
- `GameRecordService`：战绩、等级、胜率。

### 3.3 数据库与迁移

数据库相关默认高风险。

涉及以下内容必须先单独确认：

- 新增、删除或修改 entity 字段。
- 修改 repository 查询语义。
- 修改 `db/migration` 下 SQL。
- 修改生产 MySQL 或本地 H2 行为。
- 修改 Redis session 或业务缓存结构。

如需迁移，必须说明：

- schema 变化。
- 向后兼容策略。
- 本地 H2 与 MySQL 是否都适配。
- 回滚或恢复方案。

## Step 4：冒烟测试

实现后必须先做冒烟测试，再进入完整编译和测试。

冒烟测试根据改动范围选择：

| 改动范围 | 冒烟测试方式 |
| --- | --- |
| 前端 UI | 启动或使用本地前端页面，检查核心页面是否可打开、核心交互是否可用 |
| 后端接口 | 调用接口或查看启动日志，确认核心请求可成功 |
| WebSocket | 进入房间或游戏流程，确认连接、消息发送和状态广播正常 |
| 游戏规则 | 构造或运行相关测试，确认关键牌型、技能、锦囊逻辑可执行 |
| 文档 | 检查 Markdown 可读性、路径、命令和版本号是否正确 |

如果冒烟测试失败：

1. 记录失败现象。
2. 回到 Step 3 做最小修复。
3. 重新执行 Step 4。
4. 不得跳过失败直接进入 Step 5。

## Step 5：编译与自动化测试

### 5.1 后端验证

后端代码、接口、WebSocket、游戏逻辑、账号系统、社交系统、数据库访问发生变化时，执行：

```bash
cd poker
mvn test
```

如果系统没有全局 Maven，但需要使用仓库自带 wrapper，在 Windows 下执行：

```bash
cd poker
./mvnw.cmd test
```

如只改某个后端模块，应优先运行相关测试类，再运行完整测试。

### 5.2 前端验证

前端页面、状态管理、样式或构建配置发生变化时，执行：

```bash
cd poker-frontend
npm run build
```

如果依赖缺失，先确认是否允许安装依赖，再执行：

```bash
cd poker-frontend
npm install
```

不得擅自升级依赖版本。

### 5.3 全栈验证

同时涉及前后端时，至少执行：

```bash
cd poker
mvn test
```

```bash
cd poker-frontend
npm run build
```

如本地服务正在运行，完成后说明是否需要重启 Spring Boot 或 Vite dev server。

本仓库没有固定 `pm2` 流程，不使用参考文档中的 `/home/ubuntu/eval-v5.0.0` 或 `pm2 restart` 命令。

## Step 6：项目专项回归检查

完成构建和测试后，根据改动范围执行专项检查。

### 6.1 前端 UI 检查

涉及页面时必须检查：

- 桌面端主要视口。
- 手机端主要视口。
- 横屏场景，如适用。
- 弹窗、抽屉、按钮、错误提示。
- 文字是否溢出、遮挡或换行异常。
- 动画、音效、表情和图片资源是否加载正常。

### 6.2 游戏流程检查

涉及游戏逻辑时至少检查相关场景：

- 创建房间、加入房间、离开房间。
- 玩家准备、取消准备、AI 加入。
- 发牌、出牌、不要、回合切换。
- 技能触发、技能限制、觉醒或被动逻辑。
- 锦囊响应、插队、超时、结算。
- 游戏结束、战绩记录、经验等级。

### 6.3 账号与社交检查

涉及账号或社交时至少检查：

- 登录、注册、游客模式。
- 验证码、记住登录、退出登录。
- 个人中心数据展示。
- 好友申请、同意、拒绝、删除。
- 房间邀请、消息列表、未读状态。

### 6.4 WebSocket 检查

涉及实时同步时至少检查：

- 建连成功。
- 重连或刷新后状态恢复。
- 多玩家状态一致。
- 广播消息没有重复、丢失或顺序错误。
- 断线玩家、旁观者、AI 玩家状态不互相污染。

### 6.5 数据与 CRUD 检查

涉及新增、读取、更新、删除或状态持久化时必须检查：

- Controller 入参校验。
- Service 事务边界。
- Repository 查询语义。
- Entity 字段默认值和空值处理。
- DTO 不泄露敏感字段。
- 错误响应可被前端正确展示。
- H2 本地环境和 MySQL 生产环境差异。

### 6.6 编码与文件检查

中文文档、规则说明、README、SQL 注释和前端文案必须使用 UTF-8。

在 PowerShell 中读取中文文件时使用：

```powershell
$OutputEncoding = [Console]::OutputEncoding = [Text.UTF8Encoding]::new($false)
Get-Content -Encoding UTF8 -Path "<文件路径>"
```

避免因为默认编码导致中文乱码。

## Step 7：完成报告

每次更新结束后必须输出：

```md
# 更新完成报告

## 完成内容
说明实际修改了什么。

## 修改文件
列出主要修改文件。

## 需求文档
列出需求文档路径和版本号。

## 验证结果
列出执行过的命令、结果、浏览器验证或日志证据。

## 未完成或跳过项
说明原因。

## 风险提示
说明可能需要后续关注的地方。

## 规则沉淀
说明本次是否发现可复用错误模式，是否已更新 `AI_CODE_GUIDE.md`，新增或调整的规则摘要，或未沉淀原因。
```

完成报告必须明确说明：

- 是否修改了业务代码。
- 是否修改了数据库 schema。
- 是否新增或升级依赖。
- 是否存在未验证的场景。
- 是否发现仓库中已有无关改动。
- 是否需要用户手动刷新页面、重启服务或提供环境信息。
- Bug 修复任务是否完成规则沉淀判断；如未更新 `AI_CODE_GUIDE.md`，必须说明原因。

## 默认禁止行为

除非用户明确确认，否则不得：

- 修改数据库 schema。
- 新增第三方依赖。
- 升级依赖版本。
- 修改生产部署配置。
- 删除重要文件。
- 大规模调整目录结构。
- 改变认证、授权、安全策略。
- 覆盖用户已有未提交改动。
- 提交或清理大量截图、临时文件、本地数据库文件。
