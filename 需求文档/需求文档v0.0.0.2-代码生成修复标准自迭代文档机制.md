## 1. 背景

当前代码生成和修复过程存在以下痛点：

1. AI 每次生成代码时可能因为上下文不稳定导致风格不统一、边界处理遗漏或重复犯同类错误。
2. 某些模块在多轮修改中容易在同一位置反复出错，例如状态同步、响应式布局、后端空值处理、WebSocket 消息一致性、测试覆盖范围等。
3. 现有流程已经要求先写需求文档、再修改代码，但缺少一个面向“代码生成/修复标准”的长期沉淀文档，用于记录项目级规则、模块级禁忌、历史错误模式和修复经验。
4. 用户希望后续每次修 Bug 时，不只是修复当前问题，还能把本次错误原因、规避规则和验证方式融合进该标准文档，让文档具备持续自迭代能力，减少后续同类问题。

因此需要建立一个类似 skill 概念的项目内规范文档机制，使 AI 在后续改代码前优先读取并遵循该文档；当发现新错误模式时，同步更新文档，形成“问题 -> 规则 -> 验证”的闭环。

## 2. 更新目标

本次需求目标是定义一个“代码生成/修复标准自迭代文档机制”，用于指导后续实现或文档落地。

必须达成：

1. 明确该标准文档的用途：约束 AI 生成、修复、重构和验证代码的方式。
2. 明确该标准文档的优先级：后续代码修改时，应在 `WORKFLOW.md` 之后、具体源代码修改之前读取并遵循。
3. 明确该标准文档的内容结构：至少包含通用编码标准、前端标准、后端标准、测试标准、已知错误模式、修复复盘模板、自迭代更新流程。
4. 明确自迭代规则：每次修复 Bug 后，应判断是否需要把错误模式沉淀为规则；如果需要，需更新该标准文档并说明新增规则。
5. 明确边界：本次只生成需求文档，不修改业务代码，不修改已有流程文件，不创建实际标准文档。

## 5. 功能需求

### 5.1 标准文档定位

后续应建立一个项目内长期维护的“代码生成/修复标准文档”。该文档用于让 AI 在每次代码更新前快速获取项目级约束，减少不稳定生成和重复错误。

该文档应具备以下定位：

1. 不是替代 `WORKFLOW.md`，而是补充 `WORKFLOW.md` 中“如何写对代码、如何避免历史错误”的细则。
2. 不是一次性总结文档，而是随项目演进持续更新的知识库。
3. 不是泛泛的编码规范，而是结合本项目真实错误、模块结构和验证方式沉淀的可执行规则。

### 5.2 建议文档结构

后续标准文档至少应包含以下章节：

1. 文档目的与适用范围。
2. 修改前必须读取的上下文顺序。
3. 通用代码生成原则。
4. 前端 Vue/Vite 编码与样式规则。
5. 后端 Spring Boot/Java 编码规则。
6. WebSocket 与游戏状态同步规则。
7. 数据库、实体、DTO、Repository 修改规则。
8. 测试与验证最低要求。
9. 历史错误模式清单。
10. Bug 修复复盘模板。
11. 文档自迭代流程。
12. 禁止行为与高风险操作清单。

### 5.3 修改前读取规则

后续每次代码修改前，AI 应按以下顺序读取：

1. `WORKFLOW.md`
2. 项目级 AI 代码标准文档
3. 当前需求文档
4. 与任务相关的源代码、测试、配置和文档

读取后必须在回复中简要说明：

1. 本次遵循了哪些关键标准。
2. 本次可能触碰哪些风险规则。
3. 本次不会触碰哪些非范围内容。

### 5.4 Bug 修复后的自迭代规则

每次修复 Bug 后，应判断本次问题是否满足以下任一条件：

1. 属于同类错误重复发生。
2. 暴露出已有标准不够明确。
3. 涉及容易被 AI 误判的业务规则。
4. 涉及前后端状态一致性、WebSocket 消息、数据库兼容性、响应式布局、测试覆盖遗漏等高风险领域。
5. 修复过程中发现了新的“必须先检查”的上下文文件。

如果满足，应把本次经验沉淀到标准文档，内容包括：

1. 错误现象。
2. 根因。
3. 禁止的错误做法。
4. 推荐的正确做法。
5. 修改前必须检查的文件或状态。
6. 验证方式。

### 5.5 历史错误模式记录格式

建议每条历史错误模式使用统一结构：

```md
### 错误模式：{简短标题}

- 触发场景：
- 错误表现：
- 根因：
- 以后禁止：
- 推荐做法：
- 修改前检查：
- 验证方式：
- 关联模块：
- 首次记录日期：
```

### 5.6 修复完成报告增强

后续每次 Bug 修复完成报告中，应增加一个“规则沉淀”小节：

```md
## 规则沉淀

- 本次是否发现可复用错误模式：
- 是否已更新项目级 AI 代码标准文档：
- 新增或调整的规则摘要：
- 未沉淀原因：
```

如果本次 Bug 只是一次性环境问题、用户输入问题或不具备复用价值，可说明不沉淀原因。

## 6. 非功能需求

1. 文档必须使用简体中文，代码标识符、命令、日志和错误信息保留原始语言。
2. 文档必须使用 UTF-8 编码，避免 PowerShell 默认编码导致中文乱码。
3. 规则必须可执行、可检查，避免只写抽象口号。
4. 每条规则应尽量能对应“修改前检查项”或“验证方式”。
5. 文档应保持轻量，不把所有需求文档内容重复搬入标准文档。
6. 自迭代更新应只记录具有复用价值的经验，避免把一次性细节污染长期标准。
7. 标准文档更新不得替代测试和验证；规则沉淀只能减少错误概率，不能作为验收证据。

本次需求文档满足以下标准即视为完成：

1. 文档位于 `需求文档/` 目录，文件名符合 `需求文档v{版本号}-{更新名称}.md` 格式。
2. 版本号基于现有最高版本 `v0.0.0.1` 递增为 `v0.0.0.2`。
3. 文档完整包含背景、目标、任务类型、影响范围、功能需求、非功能需求、非范围、验收标准、风险与注意事项、测试计划。
4. 文档明确说明本次不修改代码。
5. 文档明确描述后续标准文档的自迭代机制。
6. 文档明确描述 Bug 修复后如何判断是否沉淀规则。
7. 文档明确说明后续标准文档与 `WORKFLOW.md` 的关系。
8. Markdown 内容可读，无明显乱码。

## 9. 风险与注意事项

1. 如果标准文档过长，AI 后续可能读取成本过高，应控制为“规则 + 检查项 + 错误模式”的结构。
2. 如果把一次性错误全部沉淀，会导致文档噪声过多，降低可执行性。
3. 如果只写原则不写验证方式，无法真正降低重复错误。
4. 如果标准文档优先级不清晰，可能与 `WORKFLOW.md`、`AGENTS.md` 或需求文档发生冲突。
5. 如果后续把“更新标准文档”变成默认必做，可能造成无意义文档膨胀；应先判断是否有复用价值。
6. 当前仓库已有大量未提交改动，本次和后续实现都必须避免覆盖用户已有工作。

## 10. 测试计划

本次仅为需求文档生成，不需要执行前端构建、后端测试或浏览器验证。

本次验证方式：

1. 检查需求文档路径和版本号是否正确。
2. 检查 Markdown 是否可读、章节是否完整。
3. 检查文档是否明确“不修改业务代码”。
4. 检查文档是否覆盖自迭代机制、错误模式记录格式和 Bug 修复后规则沉淀流程。

后续若用户确认并要求实现标准文档机制，应执行：

1. Markdown 可读性检查。
2. Git diff 检查，确认只修改约定范围内的文档文件。
3. 如修改 `WORKFLOW.md` 或 `AGENTS.md`，需复核读取顺序、确认门禁和禁止行为是否仍一致。
4. 不需要运行 `npm run build` 或 `mvn test`，除非后续同时修改业务代码。
