## 1. 背景

用户反馈：在当前对局中，当桌面上是归心玩家打出的牌，对方选择要不起，尤其是对方使用【固守】时，没有触发归心的解除禁用选择。

排查时发现，普通 `PASS` 路径已经在固守免摸牌之前调用归心待响应判断；但【固守】主动发动本身也会结束当前行动，实际等价于一次带免摸效果的要不起，却没有进入归心待响应判断。

## 2. 更新目标

- 对方在桌面牌属于禁用归心玩家时主动发动【固守】，也要先触发归心待响应窗口。
- 归心拥有者选择发动后，解除归心禁用，对方本次固守式要不起不追加摸牌。
- 归心拥有者选择不发动时，仍保留固守原本免除普通要不起摸牌惩罚的语义。
- 后端日志必须能观察归心触发检查中的桌面牌归属、归心禁用态、技能和固守保护状态。

## 3. 任务类型

问题修复 / 游戏规则状态机修复。

## 4. 影响范围

- `poker/src/main/java/com/game/poker/service/GameService.java`
  - 修复 `useGushou(...)` 漏接入归心触发判断的问题。
  - 增加归心触发检查日志，便于确认后端真实状态。
- `poker/src/test/java/com/game/poker/service/GuixinSkillServiceTest.java`
  - 新增直接发动固守也触发归心待响应并可解除禁用的回归测试。
- `AI_CODE_GUIDE.md`
  - 沉淀错误模式，避免以后只接入普通 `PASS` 而漏掉技能式要不起路径。

## 5. 功能需求

1. 当 `lastPlayedCards` 非空，`lastPlayPlayerId` 指向禁用状态的归心玩家，且当前行动玩家不是该归心玩家本人时，当前行动玩家主动发动【固守】后，必须进入 `GUIXIN_DECISION`。
2. 进入 `GUIXIN_DECISION` 时，当前行动玩家不应立即推进到下一位，需等待归心玩家响应。
3. 归心玩家接受保护后，归心解除禁用，固守保护标记被消费，回合继续。
4. 归心玩家拒绝保护后，固守保护标记仍被消费，但要不起玩家不额外摸普通 PASS 的 2 张惩罚牌。
5. 当归心不满足触发条件时，【固守】保持原有行为：摸 4 张、设置固守保护、结束行动并推进回合。

## 6. 非功能需求

- 不修改数据库 schema、认证授权、依赖版本和生产部署配置。
- 不改变普通 PASS、借刀杀人、铁骑强制清桌、观星等非本问题路径的既有语义。
- 后端仍作为归心禁用和待响应状态的唯一可信来源。
- 日志只记录技能触发判断必要字段，不输出玩家完整手牌。

## 7. 不在本次范围

- 不调整归心主动使用的摸牌数量和禁用规则。
- 不调整固守摸 4 张、固守保护标记和固守弃牌流程。
- 不新增前端交互样式。
- 不清理仓库中既有截图、本地数据库和其他无关改动。

## 8. 验收标准

1. 对方直接发动【固守】时，如果桌面牌属于禁用归心玩家，后端进入 `GUIXIN_DECISION`。
2. 后端日志出现 `【归心】触发检查`，能看到 `passer`、`tableOwner`、`ownerSkill`、`guixinDisabled`、`gushouActive`。
3. 归心玩家接受后，`guixinDisabled=false`，固守保护标记被移除。
4. 归心玩家拒绝后，固守免摸仍生效，归心保持禁用。
5. 归心相关单元测试通过。

## 9. 风险与注意事项

- 【固守】主动发动与普通 PASS 是两条后端入口，修复时必须同时覆盖。
- 如果只看前端桌面牌，不核对后端 `lastPlayPlayerId`，容易误判归心是否应该触发。
- 如果归心待响应创建后又立即 `nextTurn`，前端会看不到响应窗口，因此触发成功时必须提前 `return`。

## 10. 测试计划

```bash
cd poker
./mvnw.cmd "-Dtest=GuixinSkillServiceTest" test
```

重点验证：

- 普通 PASS 仍触发归心保护。
- 固守保护下拒绝归心仍不摸普通 PASS 惩罚牌。
- 直接发动固守也触发归心待响应，并在接受后解除归心禁用。
